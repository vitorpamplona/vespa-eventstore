Throughput vs database size

The same store logic answered three ways, all driven in-process from inside Vespa's container JVM (no client hop for any of them), at a fresh Vespa per DB size. Higher is faster; both axes are log-scaled. Six relay query shapes plus batch insert.

SQLite — Quartz in-memory DB embed — Vespa store in-process http — Vespa store over loopback
How to read it, and what flatters whom

Each point stands up a fresh single-node Vespa holding exactly that many events, so the x-axis is a true DB-size sweep, not accumulation. Numbers are single-thread throughput measured client-wall from inside the container. SQLite in-memory pays no durability and no network — it is the floor for point work and it flatters SQLite at these sizes (it erodes further out). http is loopback to the container's own port, so it omits the network a real remote Vespa would pay — the true remote gap is larger than shown. Trust ranking is excluded on both Vespa stores to isolate DB + store logic + engine transport.
